# Proyekto Mobile (Android & iOS) + Firebase Push

The mobile apps are the **same `web/` app** wrapped with [Capacitor](https://capacitorjs.com/).
The Vite build (`dist/`) is loaded inside a native WebView; there is no separate
mobile codebase. Push notifications are delivered with **Firebase Cloud
Messaging (FCM)** and reuse the existing in-app notifications domain — when any
notification row is created, the backend also pushes it to the recipient's
devices.

```
web/
├─ capacitor.config.ts      # appId tech.proyekto.app, webDir: dist
├─ android/                 # generated by `cap add android` (committed)
├─ ios/                     # generated by `cap add ios` (committed)
└─ src/
   ├─ services/pushNotifications.ts   # native FCM wrapper (no-op on web)
   ├─ services/deviceTokens.service.ts# register/unregister with backend
   └─ hooks/usePushNotifications.ts   # lifecycle, mounted in __root.tsx
```

> Web behavior is unchanged: every push path is gated on
> `Capacitor.isNativePlatform()`, and the browser keeps its existing in-app +
> Supabase-realtime notifications.
>
> The `firebase` package in `web/package.json` is only the build-time peer
> dependency of `@capacitor-firebase/messaging`'s web implementation. It is
> code-split into a lazy chunk that is never loaded at runtime (web short-circuits
> before importing it; native uses the native Firebase SDK via the bridge).
> **Browser/PWA web push is intentionally not wired** — to add it later, initialize
> the already-present `firebase` SDK and a `firebase-messaging-sw.js` service worker.

---

## 1. Prerequisites

- Node + the repo installed (`cd web && npm install`).
- **Android:** Android Studio + an SDK + a device/emulator that has **Google Play
  services** (FCM requires it).
- **iOS (macOS only):** Xcode, an Apple Developer account, CocoaPods. iOS native
  builds and signing **cannot be done on Windows** — see §6.

## 2. Firebase project setup (one-time)

1. Create / open a Firebase project at <https://console.firebase.google.com>.
2. **Add an Android app** with package name `tech.proyekto.app`. Download
   `google-services.json`.
3. **Add an iOS app** with bundle id `tech.proyekto.app`. Download
   `GoogleService-Info.plist`.
4. **iOS APNs key:** Project Settings → Cloud Messaging → *Apple app
   configuration* → upload an **APNs Auth Key (.p8)** (created in the Apple
   Developer portal: Certificates, Identifiers & Profiles → Keys). Without this,
   iOS devices never get an FCM token.
5. **Backend auth:** the backend needs permission to *send* via FCM — see §5.
   Prefer the **keyless** path (Application Default Credentials), which is also
   required if your org policy blocks downloadable service-account keys.

## 3. Place the native config files

These are **gitignored secrets** — never commit them.

| File | Location |
| --- | --- |
| `google-services.json` | `web/android/app/google-services.json` |
| `GoogleService-Info.plist` | `web/ios/App/App/GoogleService-Info.plist` (add to the Xcode target) |

## 4. Build & run

```bash
cd web
npm install                 # first time, installs Capacitor + plugins
npm run cap:sync            # vite build + copy dist into android/ & ios/
# Android:
npm run cap:open:android    # opens Android Studio → Run on device/emulator
# iOS (macOS):
npm run cap:open:ios        # opens Xcode → set signing team → Run
```

Re-run `npm run cap:sync` after any web change to refresh the native bundle.

> If the `android/` / `ios/` folders don't exist yet (fresh clone before the
> first scaffold), generate them once with `npx cap add android` and
> `npx cap add ios`, then re-add the config files from §3.

## 5. Backend Firebase credentials & env vars

The backend sends pushes via `firebase-admin`. There are two auth modes — pick
one; leaving all of them unset makes push a safe **no-op** (good for CI / local
dev without setup). Always set `CORS_ORIGINS` to include
`capacitor://localhost,http://localhost,https://localhost` so the WebView can
call the API.

### (A) Keyless — recommended (and required if org policy blocks key downloads)

Authenticates via **Application Default Credentials** — no secret to store.

```bash
# backend/.env
FIREBASE_PROJECT_ID=tech-proyekto-app
FIREBASE_USE_ADC=true
```

- **Local dev:** run `gcloud auth application-default login` once (your account
  needs FCM send access on the Firebase project — owners have it).
- **Production (Cloud Run):** the attached **runtime service account** is used
  automatically. Grant it FCM access on the Firebase project once:
  ```bash
  gcloud projects add-iam-policy-binding tech-proyekto-app \
    --member="serviceAccount:<CLOUD_RUN_RUNTIME_SA>" \
    --role="roles/firebasecloudmessaging.admin"
  ```

### (B) Service-account key — only if your org allows key downloads

```bash
FIREBASE_PROJECT_ID=tech-proyekto-app
FIREBASE_CLIENT_EMAIL=<client_email from the service-account JSON>
FIREBASE_PRIVATE_KEY="<private_key — single line, keep the literal \n>"
```
The backend un-escapes the `\n` to real newlines at runtime. If both modes are
configured, the key (B) wins.

### Production (Cloud Run) — ships dark until enabled

The deploy workflow uses the **keyless** path: when the repo variable
**`FCM_PUSH_ENABLED`** is set (any non-empty value), it injects
`FIREBASE_PROJECT_ID` (repo var, default `tech-proyekto-app`) + `FIREBASE_USE_ADC=true`
as env vars — no secret needed. So to enable in prod:

1. Grant the Cloud Run runtime SA `roles/firebasecloudmessaging.admin` on the
   Firebase project (command above).
2. Set the repo variable **`FCM_PUSH_ENABLED`** to any non-empty value.
3. Ensure the `CORS_ORIGINS` secret includes the three `localhost` origins.

Until `FCM_PUSH_ENABLED` is set, the Firebase env is not injected and deploys
stay green.

## 6. iOS — remaining manual steps (macOS + Xcode)

These can't be done on Windows; do them on a Mac:

1. Open `web/ios/App/App.xcworkspace` in Xcode.
2. Signing & Capabilities → select your **Team**.
3. Add capability **Push Notifications**.
4. Add capability **Background Modes** → check **Remote notifications**.
5. Confirm `GoogleService-Info.plist` is in the `App` target.
6. Confirm the APNs `.p8` key is uploaded to Firebase (§2.4).
7. Run on a **real device** (the iOS simulator can't receive remote push).

## 7. How the push flow works

1. On login, `usePushNotifications` requests permission, gets the FCM token, and
   `POST /api/push/tokens` upserts it into `device_tokens` (UNIQUE token, many
   rows per user → multi-device).
2. Any `NotificationsService.createNotification()` call inserts the in-app row,
   then fires an FCM push (bounded `Promise.race` timeout, errors swallowed — a
   push failure never blocks or breaks the triggering action).
3. The push payload carries `notification`(title/body) **and** `data`
   (`type`, `notification_id`, `link_url`, ids). Both are sent so a background /
   cold-start tap still has the deep-link data.
4. Tapping a notification routes to `data.link_url` (default `/notifications`)
   for foreground, background, and cold-start. Foreground receipt refreshes the
   notification queries (bell badge + lists).
5. On logout, the current device token is deleted. Tokens FCM reports as
   unregistered are pruned automatically on the next send.

## 8. End-to-end test

1. Run the app on a device, log in, and confirm a row appears in `device_tokens`.
2. Trigger a real event for that user (e.g. send them a chat mention) **or** use
   Firebase Console → Cloud Messaging → *Send test message* with the device's
   FCM token, **or** a quick admin script:
   ```js
   admin.messaging().send({
     token: "<device-fcm-token>",
     notification: { title: "Test", body: "Hello" },
     data: { link_url: "/notifications" },
   });
   ```
3. Verify all three states:
   - **Foreground** → notification queries refresh (bell updates).
   - **Background** → tray notification; tapping deep-links to `link_url`.
   - **Cold start** (app killed) → tapping launches the app and navigates to
     `link_url` (defaults to `/notifications`).
4. Log out → the `device_tokens` row is removed.

## 9. Updating the app on users' devices

A Capacitor app has **two** update layers:

| Layer | What it covers | How it updates | Needs a store release? |
| --- | --- | --- | --- |
| **Native binary** (APK/AAB, IPA) | the shell + plugins, permissions, icon, native config, Capacitor version | new build → Google Play / App Store → review/rollout | **Yes** |
| **Web bundle** (`dist/`) — OTA "Live Update" | all JS/HTML/CSS (i.e. ~all React/product changes) | device downloads the new bundle on next launch | **No** |

### Web layer — self-hosted OTA (implemented)

OTA is wired with **`@capgo/capacitor-updater`** (open-source) pointed at our **own
update server** (the backend `mobile-updates` module) with bundles stored in **R2**
(`cdn.proyekto.tech`). No paid SaaS. Flow:

1. CI (`.github/workflows/mobile-ota-deploy.yml`) runs on every merge to `main`
   under `web/**`: builds `dist/`, zips it (index.html at zip root), computes a
   sha256, **presigns** an R2 upload, PUTs the zip, and registers the bundle via
   `POST /api/mobile-updates/bundles`.
2. The app's Capgo plugin POSTs `https://api.proyekto.tech/api/mobile-updates/check`
   on foreground; the backend returns the latest published, native-compatible
   bundle (`{ version, url, checksum }`) or no-update.
3. The plugin downloads in the background and applies on the next cold start
   (`directUpdate: false`). `notifyAppReady()` (in `main.tsx`) confirms a good boot
   so a broken bundle auto-rolls-back.

> ⚠️ **The native-compatibility rule (read this).** OTA may ship **web-only**
> changes. Each bundle carries **`native_build_min`** (a native `versionCode`), and
> the check endpoint only serves a bundle to a shell whose build is `>=` it. So:
> **any change that touches native** (a new/upgraded Capacitor plugin, a new
> permission, `capacitor.config.ts`, the app icon/splash, a Capacitor/plugin
> version bump) **requires a store release**: bump the Android `versionCode` / iOS
> build, publish to the stores, and run the OTA workflow with
> `native_build_min` = the new build (via `workflow_dispatch` input). Shipping a
> web bundle that calls a missing native API to an old shell crashes at runtime.
> Pure JS/HTML/CSS changes keep the default `native_build_min` (currently `1`) →
> OTA-only, no store release.

**Enable it (ships dark until then):**
1. Generate a strong random `OTA_PUBLISH_TOKEN`. Add it to `backend/.env` (local),
   to **GCP Secret Manager** (name `OTA_PUBLISH_TOKEN`), and to **GitHub repo
   secrets** (so CI can publish).
2. Set the GitHub repo variable **`OTA_PUBLISH_ENABLED`** to any non-empty value
   (the deploy workflow then injects the secret into Cloud Run, and the OTA
   workflow stops being a no-op).

**Manual publish / rollback:**
- **Manual publish:** `workflow_dispatch` the OTA workflow (optionally set
  `native_build_min`), or curl the endpoints yourself: `presign` → `PUT` the zip →
  `bundles` (all with `Authorization: Bearer $OTA_PUBLISH_TOKEN`).
- **Rollback:** mark the bad row `UPDATE public.mobile_app_bundles SET
  status='rolled_back' WHERE version='<v>';` (devices stop being offered it), then
  **re-run the OTA workflow on the last-good commit** — it publishes a NEW
  higher-version bundle, so devices roll *forward* to good code (never served the
  rolled-back row). Never re-serve an older row (would be a downgrade).

### Native layer — store releases (manual / not yet automated)
When native changes (see the rule above), build + ship a new store binary:
- **Android** — could be automated with a GitHub Action (signed AAB → Play; needs a
  Play Console account, upload keystore, Play service-account JSON). Not set up yet.
- **iOS** — needs a **macOS runner** + Fastlane → App Store Connect (Apple Developer
  Program, signing, App Store Connect API key); gated by store review. Cannot be
  built on Windows. Not set up yet.

> First-time note: OTA only updates the web layer of an **already-installed** shell
> that contains the Capgo plugin. The first shell with the plugin must ship via the
> store; after that, web changes flow over OTA.
