# Routing & Personas

> **Last updated:** 2026-07-09 · **Status:** current

Routing is **file-based** (TanStack Router): files under
[`web/src/routes/`](../../web/src/routes/) become routes, and
`web/src/routeTree.gen.ts` is **generated by the router plugin — don't hand-edit it**.
The tree is split into per-persona/area subtrees, with auth and persona gating done in
each route's `beforeLoad`.

## Route subtrees

| Subtree | Covers |
| --- | --- |
| `auth/` | `login`, `signup`, `verify`, `callback`, `forgot-password`, `auth/admin/*` |
| `admin/` | Layout `admin.tsx` + `applications`, `match`, `approve-admin`, `settings` |
| `consultant/` | `browse`, `marketplace`, `templates`, `apply`, `$profileId` |
| `freelancer/` | `invites`, `go-live` |
| `profile/` | `profile/$profileId` |
| `teams/` | `teams/index`, `$teamId/*` (settings, time, payouts, rates), `me/invites` |
| `project/` | `$projectId` layout + tabs (below) |
| `roadmap/` | `shared/$token` (public), `shared-with-me` |

Top-level routes: `index` (landing), `dashboard`, `onboarding`, `welcome`, `inbox`,
`notifications`, `meetings`, `work-items`, `project-posting`.

> **No `client/` route subtree.** "Client" is a persona (`profile.active_persona`),
> but client-facing surfaces live under `project/` and `dashboard` — there's no
> dedicated `client/` directory.

## The project subtree

`project/$projectId.tsx` is a layout with tabs (nav in `ProjectSidebar` /
`ProjectBottomNav`, grouped Plan / Collaborate / Manage):

`overview`, `roadmap` (+ `roadmap/$roadmapId`, `roadmap/create`), `work-items`,
`team`, `payments` (shown as "Invoices"), `resources`, `logs`, `chat/$chatRef`, and
`settings/*`. There's also `project/roadmap/convert/$roadmapId` for the guest-roadmap →
project conversion.

## Auth & persona gating

Gating happens in three places:

- **`beforeLoad` auth guard (most routes)** — reads
  `useAuthStore.getState().isAuthenticated` synchronously and
  `throw redirect({ to: "/auth/login", search: { redirect } })` if not. Auth pages
  reverse-guard (already-authenticated users are bounced away).
- **Consultant verification** — `consultant/marketplace` and `consultant/templates`
  add a `beforeLoad` check that redirects to `/dashboard` unless
  `profile.is_consultant_verified`.
- **Component guards** — finer-grained access is enforced in components:
  `RequireProjectAccess` (backed by `project_access` role) wraps project bodies, and
  `ProtectedRoute` supports a `requiredPersona` check against `active_persona`. Admin
  gating runs in the `admin.tsx` layout via an `adminMe` query (shows "Access Denied"
  if not an admin).

> The **`projectId === "n"`** sentinel is the guest / roadmap-only path — its
> `beforeLoad` skips the auth check so guests can build a roadmap before signing up.
> See [Feature Domains → guests](../11-domains/guests.md).

## Adding a route

1. Add a file under `web/src/routes/` (the plugin regenerates `routeTree.gen.ts`).
2. Add a `beforeLoad` auth guard if it's authenticated.
3. Wrap in `RequireProjectAccess` / check persona if it's access-scoped.
4. If it's a new top-level page, remember to keep the header/nav's known-paths in
   sync (per the web theme conventions).
